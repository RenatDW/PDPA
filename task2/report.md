# Отчет по лабораторной работе

## Тема

Параллельное вычисление `trace(A x B)` для матриц размера `N x N` на C++ с использованием OpenMP.

## Постановка задачи

Требуется реализовать последовательную и параллельную версии вычисления:

$$
\mathrm{trace}(A \cdot B)=\sum_{i=0}^{N-1}(A\cdot B)_{ii}=\sum_{i=0}^{N-1}\sum_{k=0}^{N-1}A_{ik}B_{ki}
$$

В параллельной версии использовать блочный подход: каждый поток обрабатывает свой набор блоков и добавляет локальную сумму в общую переменную с применением `locks`.

Также требуется:

1. Сравнить время выполнения для разных политик планирования OpenMP: `dynamic`, `guided`, `static`.
2. Для каждого набора запусков вычислить минимальное, максимальное и среднее время.
3. Проверить корректность результата (одинаковый `Trace(A*B)` для всех запусков).

## Цель работы

Изучить влияние параметров OpenMP (тип `schedule` и число потоков) на производительность параллельного алгоритма вычисления следа произведения матриц.

## Краткое описание решения

1. Матрицы `A` и `B` хранятся в одномерных массивах длины `N*N`.
2. Используется блочная схема обхода с размером блока `TILE = 32` для повышения локальности данных.
3. Внутри параллельной области каждый поток накапливает значение в `threadSum`.
4. После завершения своей части работы поток добавляет `threadSum` в общую сумму `tracePar` внутри критического участка на `omp_lock_t`.
5. Время измеряется через `omp_get_wtime()`.

## Параллельная версия (OpenMP + locks)

В рабочей версии используется следующая схема:

```cpp
__int128 tracePar = 0;
omp_lock_t lock;
omp_init_lock(&lock);

#pragma omp parallel
{
   __int128 threadSum = 0; // private для потока

   #pragma omp for collapse(2) nowait schedule(...)
   for (int ii = 0; ii < n; ii += TILE)
      for (int kk = 0; kk < n; kk += TILE) {
         int iEnd = std::min(ii + TILE, n);
         int kEnd = std::min(kk + TILE, n);
         for (int i = ii; i < iEnd; ++i)
            for (int k = kk; k < kEnd; ++k)
               threadSum += 1LL * A[i * N + k] * B[k * N + i];
      }

   omp_set_lock(&lock);
   tracePar += threadSum;
   omp_unset_lock(&lock);
}

omp_destroy_lock(&lock);
```

## Обоснование shared и private переменных

`shared`:

1. `A`, `B` - общие входные массивы; потоки их только читают.
2. `tracePar` - общая итоговая сумма, в которую каждый поток добавляет локальный результат.
3. `lock` - общий объект синхронизации, обеспечивающий взаимоисключение при обновлении `tracePar`.
4. `N`, `n`, `TILE` - параметры задачи, одинаковые для всех потоков.

`private`:

1. `threadSum` - локальный аккумулятор потока, чтобы избежать гонок и уменьшить число синхронизаций.
2. Индексы циклов (`ii`, `kk`, `i`, `k`) - по правилам OpenMP являются приватными для итераций цикла.
3. Временные границы блока (`iEnd`, `kEnd`) - вычисляются отдельно в каждой итерации.

Почему именно так:

1. Если сделать `threadSum` общим, появится гонка при суммировании.
2. Если обновлять `tracePar` на каждой операции умножения, синхронизация станет слишком частой и замедлит программу.
3. Локальная сумма + одно защищенное добавление на поток дает хороший компромисс между корректностью и производительностью.

## Минимальное, максимальное и среднее время

### По каждому типу schedule

| Schedule |  Tmin, с |  Tmax, с |  Tavg, с |

### По всем выполненным запускам

$$
T_{\min}=0.877703\;\text{с},\quad
T_{\max}=1.49538\;\text{с},\quad
T_{\text{avg}}=\frac{\sum T_i}{9}=1.101327\;\text{с}
$$

## Анализ результатов

1. Наблюдается снижение времени с ростом числа потоков для всех политик планирования.
2. Лучший результат показал `guided` на 24 потоках (`0.877703` с).
3. На данном наборе тестов `dynamic` оказался самым медленным.
4. Ускорение при переходе с 8 на 24 потока:

$$
S_{dynamic}=\frac{1.49538}{1.04547}\approx1.43,\;
S_{guided}=\frac{1.19610}{0.877703}\approx1.36,\;
S_{static}=\frac{1.20155}{0.920308}\approx1.31
$$

## Вывод

В рамках задачи №22 реализован параллельный блочный алгоритм вычисления `trace(A x B)` на C++/OpenMP с использованием `locks` для объединения частичных сумм потоков. Корректность подтверждена одинаковым значением `Trace(A*B)` во всех экспериментах.

По предоставленным данным наиболее эффективной политикой планирования является `guided` (минимальное среднее и минимальное абсолютное время). Выбор `shared` и `private` переменных обеспечивает отсутствие гонок данных и приемлемые накладные расходы на синхронизацию.

Примечание по методическим требованиям: в текущих измерениях использованы 8, 16 и 24 потока. Для полного соответствия формулировке лабораторной обычно дополнительно проводят тесты на 2, 4, 6, 12 и 24 потоках и подбирают `N` так, чтобы последовательная версия работала около 30-120 секунд.
